# Plano de Análise: Erro de Tipo em objects.slynx

## Problema
O código `pessoa.age = f().age;` na linha 25 do arquivo `objects.slynx` gera erro:
- Esperado: `Reference { rf: TypeId(8), generics: [] }`
- Recebido: `Struct { fields: [] }`

## Contexto
1. `f()` retorna `Person`, que é um alias de `Pessoa` (linha 10: `alias Person = Pessoa`)
2. O usuário reportou que no HIR (linha 321-357 de `hir/mod.rs`), ao usar `retrieve_declaration_data_by_name`, o tipo é `TypeId(9)`

## Análise da Causa Raiz

### Possível problema 1: Debug prints suspeitos
Em `frontend/src/hir/mod.rs:346-347`:
```rust
println!("{:?}", self.types_module.get_type(&target_ty));
println!("{ty:?} {:?}", self.types_module.get_type_mut(&ty));
```
Esses prints indicam que houve manipulação do tipo durante o processamento do alias. O código modifica o tipo do alias para um Reference mas pode estar causando problema na resolução.

### Possível problema 2: Resolução do tipo de retorno de função
Quando `get_function_signature` é chamado (`checker/expr.rs:316`), ele busca o tipo da função na tabela `declarations`:
```rust
fn get_function_signature(&self, declaration: DeclarationId, span: &Span) -> Result<(Vec<TypeId>, TypeId)> {
    let Some(function_ty) = self.declarations.get(declaration.as_raw() as usize).copied()
```
O tipo de retorno da função pode não estar sendo resolvido corretamente para o alias `Person`.

### Possível problema 3: Tipo da expressão FieldAccess
Em `get_type_of_expr`, quando processa `f().age`:
1. Primeiro avalia `f()` → deveria retornar `Reference(Pessoa)` 
2. Depois avalia `.age` → chama `resolve_field_access_type`

Se o tipo de `f()` não for resolvido corretamente como Reference, o `FieldAccess` terá tipo Struct vazio.

## Passos para Diagnóstico

1. **Remover debug prints**: Os prints em `hir/mod.rs:346-347` indicam código de debugging que deve ser removido ou analisado

2. **Verificar tipo da função f()**: No HIR, qual o TypeId do tipo de retorno da função `f`? Deve ser o tipo do alias `Person` (TypeId 9)

3. **Verificar resolução do alias**: No checker, quando `resolve()` é chamado no tipo de retorno de `f()`, ele deveria resolver o Reference para o Struct de Pessoa

4. **Adicionar logging de debug**: Durante o type checking de `f().age`, imprimir:
   - Tipo de retorno de `f()` 
   - Tipo após `resolve_field_access_type`
   - Tipo após `resolve()`

## Hipótese Principal
O problema está na forma como o tipo de retorno da função `f()` é armazenado ou resolvido. O alias `Person` está sendo armazenado com tipo `Struct { fields: [] }` em vez de manter a referência ao tipo original `Pessoa` que tem campos.

## Ação Recomendada
Investigar como o tipo de retorno da função é definido no HIR quando a função retorna um alias, e garantir que o tipo de retorno preserve a referência ao tipo real (Reference) em vez de ser resolvido para o tipo base prematuramente.
